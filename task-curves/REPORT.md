# Отчёт

## Два подхода:

**Подход 1 - загрузка LUT из файла во время выполнения**

**Подход 2 - LUT, скомпилированный в бинарник (hardcoded):**

---

## Замеры времени (изображение 1920×1080, 100 итераций)

| | Подход 1 (из файла) | Подход 2 (hardcoded) |
|---|---|---|
| Применение LUT, мс/кадр | ~0.425 | ~0.423 |
| Загрузка LUT при старте | чтение + декодирование PNG | 0 (данные уже в памяти) |
| Max pixel diff | — | 0 (результаты идентичны) |

---

## Почему Подход 2 быстрее

Время применения `cv::LUT()` у обоих подходов практически одинаково (~0.42 мс/кадр), операция поиска по таблице одна и та же. Принципиальное различие в фазе инициализации:

- **Подход 1** при каждом запуске выполняет `cv::imread()`: читает PNG-файл с диска, декодирует его, только потом строит LUT. Это io, которая зависит от скорости диска и загруженности ос

- **Подход 2** имеет LUT-массивы, вшитые прямо в исполняемый файл на этапе сборки. При старте программы данные уже находятся в памяти процесса — никакого I/O нет
